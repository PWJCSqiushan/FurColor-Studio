# FurColor Studio 完整操作手册

本文说明 Windows 本地完整版从安装、建项目、选片、隐私复核、眼睛标注、渲染到客户交付的完整流程。第一次使用建议从头到尾走一遍，不要跳过分析和人工复核。

## 1. 安装

### 1.1 自动安装

在 PowerShell 中执行：

```powershell
git clone https://github.com/PWJCSqiushan/FurColor-Studio.git
Set-Location '.\FurColor-Studio'
powershell -ExecutionPolicy Bypass -File '.\install_local.ps1' -AllowedRoots 'D:\Photography' -DownloadFaceModel -Launch
```

把 `D:\Photography` 替换为你的照片根目录。允许多个目录：

```powershell
-AllowedRoots 'D:\Photography;E:\Events'
```

安装器不会扫描照片，只会把这些目录记录成本地访问白名单。安装完成后，原生“浏览”按钮还可以动态授权更多目录：

- 原片、参考、分析、输出、水印和清单可以位于不同磁盘；
- 每次选择的目录都会写入本机 `runtime/authorized_roots.json`；
- 移动硬盘盘符变化后重新选择即可；
- 动态授权必须由本机原生选择器和临时本地令牌共同完成；
- `.ssh`、`.aws`、`.git`、AppData、credentials 等敏感目录始终禁止访问。

### 1.2 安装器做了什么

- 创建 `.venv`，不污染系统 Python；
- 安装 `requirements.txt` 中锁定的依赖；
- 创建被 Git 忽略的 `.env`；
- 下载固定版本的 OpenCV YuNet；
- 计算并比对 SHA-256；
- 检查 Python 模块、白名单目录和模型文件；
- 使用浏览器打开本地 UI。

安装器不会上传照片、写入相机原片或连接 FurColor 云端。

### 1.3 启动与诊断

```powershell
& '.\run_local.ps1'
```

如果启动失败：

```powershell
& '.\doctor.ps1'
```

### 1.4 启用 Fursee 4.0 主体智能层（可选）

基础 `.venv` 继续负责 UI、YuNet、选片和渲染；Fursee 使用独立 `.venv-fursee`，避免 PyTorch/Transformers 改坏现有环境。取得完整模型包后运行：

```powershell
powershell -ExecutionPolicy Bypass -File '.\install_fursee.ps1' `
  -ModelDirectory 'D:\Models\Fursee-Package'
```

模型目录可以位于其他磁盘。安装器只引用原位置，不复制权重；完成前会核对四个模型文件的固定尺寸和 SHA-256。默认安装 CUDA 12.8 版本，首次下载约数 GB。没有 NVIDIA GPU 时可使用 `-CpuOnly`，但不建议处理数百张活动照片。

诊断：

```powershell
& '.\doctor.ps1'
```

看到 `Fursee imports: OK`，并在工作台看到“Fursee 就绪”，才表示主体智能层可用。
## 2. 活动开始前规划目录

每个活动使用独立目录，避免不同项目的参考样片、分析和交付互相污染：

```text
2026-Event\
├─ Originals\
├─ References\
├─ Analysis\
├─ Drafts\
└─ Delivery\
```

规则：

1. `Originals` 只放相机文件或伴随 JPG，永远不作为输出目录；
2. `References` 只放你确认满意的手修 JPG；
3. `Analysis` 和 `Drafts` 不得位于 `Originals` 内部；
4. 客户交付前先在 `Drafts` 完成人工检查；
5. 水印源 PNG 放在独立品牌资产目录，不要放进 Git 仓库。

## 3. 准备手修参考照片

FurColor 不是凭空猜测个人风格，它需要你手修的参考 JPG。

### 3.1 如何挑参考图

优先覆盖本次活动的主要灯光和主体：

- 暖色会场灯光；
- 冷色或蓝紫舞台灯；
- 白毛主体；
- 黑毛主体；
- 正面、侧面和逆光；
- 不同区域或灯光发生切换的位置。

建议每个明显不同的灯光场景至少准备 1–3 张参考图。参考越有代表性，邻近照片的白平衡、曝光和色彩映射越稳定。

### 3.2 命名要求

手修 JPG 与原片必须同名：

```text
Originals\DSC01151.ARW
References\DSC01151.jpg
```

项目创建时可以不提供 JSON 清单。系统会扫描 `References`，寻找同名原片并自动生成基础清单。如果没有任何匹配项，项目会拒绝创建并提示检查命名。

### 3.3 Lightroom 导出建议

- 使用完整分辨率或足够大的 JPG；
- 使用 sRGB；
- 不要在文件名后增加 `_edit`、`-final` 等后缀；
- 可以包含裁切和全局调色；
- 如果做了复杂局部修饰，应在最终草稿中额外检查模型是否错误学习局部差异。

## 4. 创建项目

点击左侧“新建项目”。

### 4.1 项目名称

建议使用活动名和日期，例如 `2026-07 Event A`，方便将来审计交付。

### 4.2 原片目录

支持 JPG、JPEG、ARW。存在同名 JPG/ARW 时，选片缩略图优先使用 JPG，提高浏览速度；最终引擎仍可读取 ARW。

### 4.3 手修参考目录

必须包含至少一组同名参考。这个目录决定模型学习哪一种会场风格。

### 4.4 分析结果目录

保存：

- 每张照片的分析配方；
- 人脸候选和风险级别；
- 质量指标；
- 处理中间结果。

这里包含隐私派生数据，不应上传公开仓库。

### 4.5 修图输出目录

保存 AI 草稿、波形图和分类输出。它必须与原片目录完全分离。

### 4.6 水印 PNG

可选。建议透明背景、边缘留白少、尺寸足够。当前默认：

- 透明度约 80%；
- 宽度约为画面宽度 6%；
- 距边约为画面尺寸 1.5%。

每个项目都可以选择不同水印。

### 4.7 参考样片清单

- 普通用户：留空，自动生成；
- 高级用户：选择 Lightroom 分析导出的 JSON，保留更详细的显影参数。

### 4.8 选片模式

| 模式 | 默认照片是否进入流程 | 适合场景 |
|---|---:|---|
| 反选 | 是 | 成片率高，只需淘汰少数废片 |
| 正选 | 否 | 连拍多，需要严格精选 |

## 5. 选片

创建项目后系统扫描原片并进入选片台。

- “保留”：明确进入后续流程；
- “废片”：从分析、眼睛标注、渲染和交付全部排除；
- “默认”：恢复当前正选/反选规则。

优先淘汰：

- 明显失焦和运动糊；
- 完全重复的动作；
- 主体严重遮挡；
- 构图无法修复；
- 出现未获授权真人脸且不值得处理；
- 胸牌、二维码或其他隐私无法安全移除。

选择会即时保存。刷新页面或重新启动后不需要重新选片。

## 6. 兽装主体分析

启用 Fursee 后，在“兽装分组”或“处理流程”中点击“分析主体”。首次任务会完整核验权重，随后对当前有效选片执行：

1. YOLO 双类别检测：兽装头部与新增真人脸候选；
2. 保存项目私有的兽头裁剪预览；
3. DINOv3 将每个兽头转换为 512 维归一化特征；
4. DBSCAN 在当前项目内生成 `C001`、`C002` 等匿名分组；
5. 写入项目私有 `subject_analysis.json` 与 `subject_embeddings.npz`。

匿名分组适合快速查看同一兽装的连拍，并可“整组保留 / 整组废片 / 恢复默认”。它不是实名识别：

- 不记录角色名、兽主姓名或联系方式；
- 不跨项目搜索或关联；
- 不映射真人身份；
- 单独出现或相似度不足的主体保留为未分组，不强行归类；
- 分组错误时应回到逐张选片，不把分组当作事实标签。

主体分析不是强制步骤。模型环境不可用时，基础 YuNet 和 V3.3 兼容流程仍能运行，但不会获得双模型误检抑制、匿名分组与兽头局部曝光。
## 7. 隐私与参考分析

点击“开始分析”。这一阶段会：

1. 读取有效选片；
2. 为目标照片寻找相近手修参考；
3. 生成 Lightroom 风格配方；
4. 估计中性区域、曝光和高光风险；
5. 使用 YuNet 产生真人脸候选；
6. 应用已积累的本地人脸记忆；
7. 若主体分析存在，融合 Fursee 兽头框与人脸候选：强一致的兽头证据可抑制耳朵误检，Fursee 单独发现的人脸只进入人工复核；
8. 写入 `Analysis\recipes`。

任务日志出现完成信息后再进入下一步。若提示找不到模型，重新执行：

```powershell
& '.\install_local.ps1' -DownloadFaceModel
```

## 8. 人脸人工复核

点击“打开人脸复核”。检测器只提供候选，摄影师需要判断：

- 真实人物正脸、侧脸和局部脸；
- 镜子、玻璃、手机和屏幕反射；
- 兽装耳朵、眼睛图案是否被误检；
- 海报和屏幕中的印刷人脸；
- 胸牌、群号、二维码等模型未覆盖的隐私。

把候选标记为：

- `human`：真人脸或需要按真人隐私处理的内容；
- `fursuit`：兽装结构或其他误检。

当两类各至少积累 2 个样本时，系统自动更新项目私有逻辑回归校准器。训练准确率不是验证准确率；换会场、换灯光、换兽装后仍要继续复核。

## 9. 逐图标注兽装眼睛

点击“标注兽装眼睛”。界面只加载当前有效选片。

每张照片需要单独标注，因为以下因素都会改变眼睛形状：

- 焦段；
- 拍摄距离；
- 头部旋转和俯仰；
- 透视关系；
- 不同兽装的眼睛造型。

蒙版应覆盖瞳孔/眼睛主体，避免跨到毛发、眼眶外沿或高光区域。V4 默认做轻量曝光和饱和度增强，最终仍需检查边缘是否自然。

## 10. V4 渲染

点击“开始渲染”。处理顺序大致为：

1. 参考样片颜色映射；
2. 中性区域白平衡校正；
3. 全图曝光、高光肩部压缩与阴影保护；
4. 根据兽头框单独比较主体与场景亮度，做最多约 ±0.38 EV 的羽化局部补偿；
5. 阴影、纹理、清晰度和色彩处理；
6. 眼睛局部增强；
7. 隐私分类输出；
8. 水印；
9. JPEG 和逐图质量指标。

输出通常按隐私风险进入不同目录，例如：

- `face_high`：高风险人脸，必须复核；
- `manual_review`：模型不确定，需要人工判断；
- `no_face_detected`：未检测到真人脸，但仍不能跳过全量快速检查；
- `quality_metrics`：质量与曝光分析；
- `waveforms`：辅助判断曝光的波形图。

## 11. 渲染后检查

### 白平衡

- 检查不过曝的白毛、白墙和中性灰；
- 不要把本来有色的舞台光强行校成白色；
- 同一场景相邻照片应保持一致；
- 灯光切换处增加抽检。

### 曝光

- 白毛高光是否失去纹理；
- 黑毛是否压成一团；
- 头部和眼睛是否明显欠曝；
- 主体是否因背景高亮而被整体压暗；
- 查看波形图，但以主体观感和细节为最终依据。

### 眼睛与蒙版

- 增强是否落在正确眼睛；
- 是否越过轮廓；
- 饱和度是否过高；
- 左右眼强度是否自然。

### 隐私与水印

- 背景真人脸和反射；
- 兽耳误打码；
- 胸牌、二维码和屏幕内容；
- 水印是否过大、被裁掉或遮挡主体。

## 12. 交付

在“质检与交付”页完成四项确认。前端和后端都会检查确认状态。

生成内容：

- 最终 JPG；
- `manifest.json`；
- 每个文件的 SHA-256；
- ZIP 压缩包。

建议交付前再使用系统图片浏览器做一次全量快速浏览，而不只检查模型标记的风险图。

## 13. 中断、继续和重跑

- 选片状态保存在本地 SQLite 中；
- 人脸标签和记忆模型保存在项目私有运行目录；
- 眼睛标注按文件名保存；
- 重新渲染不会修改原片；
- 如需保留旧草稿，重跑前把旧输出目录改名或在新项目中使用新输出目录。

## 14. 数据与 Git 安全

禁止提交：

- `.env`；
- JPG/JPEG/ARW/PNG；
- ONNX 模型；
- SQLite 数据库；
- 人脸反馈 JSONL；
- 项目级 `subject_analysis.json`、兽头裁剪与 `*.npz` 特征向量；
- 眼睛坐标；
- 水印和客户交付 ZIP；
- SSH 私钥。

公开前运行：

```powershell
& '.\.venv\Scripts\python.exe' scripts\security_audit.py
```

必须看到 `PASS`。
